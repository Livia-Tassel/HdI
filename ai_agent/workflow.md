# AI Agent Collaboration Workflow

## 说明
本文件记录 HdI 项目在本地开发阶段的真实 AI 协作流程，覆盖赛题理解、数据结构辨识、代码重构、调试、结果解释与交付整理。项目遵循“数据优先、结果诚实、材料可复现”的原则：先按赛题 provided 数据的真实结构重建分析链路，再组织报告、静态 API、英文仪表盘与 AI 协作材料。

## 本轮协作阶段

### 1. 赛题理解与交付约束确认
- 重新阅读 `resources/instruction.pdf`，确认最低提交材料包括：
  - 主题分析报告 PDF
  - 数据处理源程序
  - 作品展示视频
  - 自增数据集（若有）
  - 智能体交互过程记录
- 明确本地实施范围：
  - 报告、脚本、静态数据、仪表盘和 AI 协作材料全部补齐
  - 视频保留脚本与演示路径，录制动作由团队后续完成
  - 默认不把未落地的探索性方法写入最终结论

### 2. 真实数据结构辨识
- AI 协助检查 `data/raw/provided` 下 5 类数据的真实目录结构、列名、年份范围与指标形态。
- 识别出原脚手架的关键不匹配：
  - 原代码将若干中文列名正则清洗为空列
  - 原 country harmonization 会把部分中文国家名误判成合法 ISO3
  - 原 notebook、前端与 API 中存在虚构的 `daly_rate`、`GHRI`、`load_daly` 等旧假设

### 3. 数据层重构
- AI 协助设计并实现按数据源拆分的清洗器与整合器：
  - 死因数据清洗与三大类映射
  - 风险因素归因死亡清洗
  - HNP/WDI 长表提取与指标透视
  - 中国省级卫生数据长表化
- 国家映射采用 World Bank 元数据、Babel 中文地名与手工别名补丁，稳定输出 ISO3。
- 构建并验证核心面板：
  - `data/processed/master_panel.parquet`
  - `data/processed/resource_panel.parquet`
  - `data/processed/china_panel.parquet`

### 4. 三个分析维度落地
- **Dimension 1**
  - 用真实死亡原因数据构建“传染性疾病 / 非传染性疾病 / 伤害”三大类长期趋势
  - 用预期寿命、死亡率、卫生资源与社会经济指标完成解释与基线预测
- **Dimension 2**
  - 基于 provided 风险归因死亡结果做“归因死亡贡献分析”
  - 保留 `/dim2/paf` 兼容接口，但在结果元数据中明确主方法是 `attributable_share`
- **Dimension 3**
  - 用资源投入指数、健康产出指数和需求指数构建资源缺口与效率象限
  - 用数学规划模拟跨国资源再分配
  - 用中国省级数据补充国内资源差异视角

### 5. 扩展功能：Optimization Lab
- 在赛题要求之外，AI 协助将 Dimension 3 再分配模块扩展为可比较的“Optimization Lab”。
- 该扩展围绕不同优化目标与预算假设组织静态场景结果，目的是让英文仪表盘能够离线展示“当前配置、优化方案、主要受益方和主要让渡方”的对比。
- 与主线结果的关系是：
  - 不引入新的原始数据源
  - 继续复用 Dimension 3 的资源-产出-需求框架
  - 作为主题相关的展示增强，而不是独立第四维度

### 6. 前端与导出链路调试
- AI 协助修复和收敛英文静态仪表盘：
  - 去除非必要远程依赖，改为本地 Plotly
  - 移除脆弱的启动遮罩与远程旗帜资源
  - 统一地图、国家详情、风险排行与 Optimization Lab 的状态切换
  - 为 spotlight 面板补齐可访问性对话框语义
- 同步修复静态 API 与导出契约：
  - `/api/v1/dim3/optimization` 正确支持 `objective` 与 `budget_multiplier`
  - `/api/v1/dim2/shapley` 对未实现的 `disease` 过滤显式标注“不支持”

### 7. 提交材料整理
- 自动导出报告图表和表格至 `reports/figures/` 与 `reports/tables/`
- 重写 `video/demo_script.md`，确保演示脚本只描述真实产物
- 更新：
  - `data/catalog.yaml`
  - `reports/report.tex`
  - `ai_agent/submission_record.md`
  - `ai_agent/log_index.json`

## 智能体在本项目中的具体贡献
1. **需求理解**：将赛题说明拆解为三大分析维度和五类提交物。
2. **方法取舍**：帮助区分“可落地方法”和“仅探索过的方法”，避免把未实现的 GHRI/DEA 等方案写入最终主线。
3. **代码生成与重构**：完成清洗器、整合器、分析导出脚本、静态 API 结果导出与英文仪表盘重构。
4. **调试**：定位并修复中文列名清洗、国家代码误判、接口参数失真、前端远程依赖与对话框可访问性等问题。
5. **结果解释与包装**：提炼三大维度结论，组织报告、视频脚本、仪表盘文案和 AI 协作材料。

## 交互证据与说明
- `ai_agent/submission_record.md`
  - 面向比赛提交的协作记录摘要，说明 AI 在需求理解、方法选择、代码与调试、结果解释中的具体贡献。
- `ai_agent/log_index.json`
  - 将现有 JSON 日志标注为“已采纳 / 部分采纳 / 仅探索”，避免把早期方案误读为最终结果。
- `ai_agent/interaction_logs/*.json`
  - 早期对话样例，主要体现方法探索与架构讨论。
  - 其中部分方法（如 GHRI、DEA 等）未进入最终交付，应视为“探索记录”，而不是最终结果来源。
- `ai_agent/workflow.md`
  - 当前文件，描述从真实数据到最终交付的完整协作流程。
